# 端到端範例：從收到需求到完成 CI

## 範例題材
**為轉帳 API 加入「每日累積轉帳上限」驗證**
（小到能具體說明，但受法規約束，足以展示安全迴圈的價值）

這個範例對應一頁題目中的 Actor↔Critic 受控迴圈：執行者（Codex 寫碼）與評審者（QA agent、Security agent）互相搭配，以「規格為目標、憲法與測試為安全把關」自動迭代到達標，合併前再交人工確認。

---

## 角色與工具對照

| 角色 | 由誰扮演 | 使用工具 |
| --- | --- | --- |
| Spec agent（規格） | 自訂 GPT | 一般對話 + Connector 讀取需求單 |
| Actor（執行者，寫碼） | Codex | Connector 串接程式庫（讀寫分支） |
| QA Critic（驗收） | 自訂 GPT | Connector 觸發 CI 跑測試、回讀結果 |
| Security Critic（憲法檢核） | 自訂 GPT | 唯讀存取程式碼 + 憲法規則集 |
| Orchestrator（迴圈控制） | App 既有的編排器 | 管理迭代次數、時間上限、人工關卡 |

> **「憲法」定義**：團隊事先以文字明訂、且**版本控管、可持續維護**的一組安全與合規規則（例如：金額一律用十進位定點數、所有金流操作必須留稽核軌跡、不得在日誌輸出顧客個資、外部輸入一律先驗證）。它是 Security Critic 檢核的依據，對寫碼的 agent 唯讀、不可竄改。**新增或修改一條條款，安全網就立即套用到之後所有的變更**——規則活在檔案裡，不在個人記憶或散落的程式碼中。此概念源自 spec-driven development（SDD）的 constitution，我們的延伸是替它補上「自動強制檢核」這一層。

---

## 完整流程：七個階段

### 階段 0 — 收到需求
產品經理在需求系統建立一張單：「轉帳前需檢查當日累積金額是否超過顧客等級上限，超過則擋下並回傳明確錯誤。」

Orchestrator 透過 **Connector** 讀取這張單，啟動一輪迴圈，並設定本輪上限（例如：最多 6 次迭代或 20 分鐘）。

### 階段 1 — Spec agent 把需求轉成可驗收規格
Spec agent（自訂 GPT）收到的提示語大致是：

> 你是規格分析師。根據以下需求單，產出（a）驗收情境清單，每條都要可被自動測試；（b）本功能適用的憲法條款編號。只輸出 JSON，不要說明文字。

輸出範例（節錄）：
- AC-1：當日累積 + 本次金額 ≤ 上限 → 放行
- AC-2：當日累積 + 本次金額 > 上限 → 擋下，回傳錯誤碼 `LIMIT_EXCEEDED`
- AC-3：金額計算一律用定點數，不得用浮點數（憲法 §3）
- AC-4：擋下事件必須寫入稽核軌跡（憲法 §5）

這一步把「人話需求」變成「機器可驗證的靶」，是後面所有自動迭代的依據。

### 階段 1.5 — 人工關卡 #1：審查規格（寫碼前）
在把規格交給 Actor 之前，由人先審查這份「靶」是否對準業務真意。三種結果：**核准 / 修改 / 退回**。

> 為什麼這一關不能省、也無法自動化：後面的測試只能查「程式對不對規格」，Security Critic 只能查「程式對不對憲法」，但**沒有任何自動關卡能查「規格對不對需求」**。若 Spec 把「當日累積上限」誤解成「單筆上限」，整個迴圈會忠實地把錯的功能寫好、測過、開成 PR——每一關都綠燈。把需求層級的錯誤擋在這裡，是最便宜的。

成本很低：規格**每個功能只審一次**（不是每輪都審），而且讀的是一份簡短的結構化規格、不是程式碼。昂貴的內層迴圈（寫→測→修）仍然無人值守地自動跑。一句話：**人審「靶」，機器負責「命中」。**

### 階段 2 — Actor（Codex）寫第一版
Codex 收到規格與相關程式碼脈絡，提示語重點：

> 你是執行工程師。實作以下驗收情境。可以讀取程式庫，但**不可修改任何測試檔或憲法檔**。完成後在 `feature/transfer-limit` 分支提交。

Codex 透過 **Connector** 讀取現有轉帳模組、寫出實作與對應單元測試，推到功能分支。

### 階段 3 — QA Critic 跑驗收（確定性把關優先）
QA Critic 透過 **Connector 觸發 CI**，跑完整測試套件與階段 1 的驗收情境，回讀結果。

- 若有測試失敗 → 把失敗訊息與堆疊摘要交回 Actor，進入下一輪（階段 2）。
- 重點設計：**先用確定性的測試把關，再交給 LLM 評審**。能用測試明確判定對錯的，就不浪費在 LLM 主觀判斷上。

假設第一輪 AC-2 失敗（邊界值剛好等於上限時誤擋），訊息回到 Codex，Codex 修正比較邏輯，重提。

### 階段 4 — Security Critic 對照憲法
測試全綠後，Security Critic（自訂 GPT，**唯讀**）才上場。提示語重點：

> 你是安全審查員。比對以下變更與憲法條款。逐條回答「符合 / 違反 / 不適用」並附證據行號。你只能讀，不能改任何檔案。

它可能抓到：金額用了浮點數型別（違反 §3）→ 退回 Actor 修正為定點數型別。這類問題測試不一定抓得到，正是 LLM 評審補上的價值。

### 階段 5 — 收斂或交回人工
迴圈在「驗收情境 ＋ 憲法 ＋ 測試」全通過時收斂；或達到迭代/時間上限時**停止並交回人工**，附上目前狀態與卡關原因。不會無限跑。

### 階段 6 — 人工確認後合併（強制關卡）
通過所有自動關卡後，Orchestrator 產出一個**已附測試結果與憲法檢核報告的 PR**。工程師做最後人工確認才能合併——這一關不可略過。合併觸發 CI 進入部署流程。

## 防弊設計（直接對應評分的「可行性」）

| 風險 | 對應的迴圈設計 |
| --- | --- |
| Reward hacking（為過測試偷改測試） | 驗證機制（測試＋憲法）對 Actor **唯讀、不可竄改** |
| 無限迴圈 / 失控 | 設**迭代次數與時間上限**，達標即停或交回人工 |
| LLM 主觀誤判 | **先確定性測試**，再 LLM 評審，分工把關 |
| 違反合規而不自知 | Security Critic 逐條對照**憲法**，附證據 |
| 自動合併出錯 | 合併前**一定**要人工確認 |
| 建錯東西（規格誤解業務真意） | **人工關卡 #1**：寫碼前先審查規格；此判斷自動關卡無法代勞 |

---

## 一句話收束
這個範例證明：交付的不是一個會失控自跑的 AI，而是一條「設有上限、結果可驗證、寫碼與評審分離、驗證不可被竄改、**且在規格與合併兩端都有人工把關**」的受控迴圈——人決定「做對的事」（審規格）與「接受這次變更」（審 PR），機器負責中間的寫、測、修。同樣的迴圈紀律已設計進我們 App 的 orchestrator，並以此 App 作為實證。
